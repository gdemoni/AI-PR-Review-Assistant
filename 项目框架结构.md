## 3. 后端架构详解

### 3.1 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| Web 框架 | FastAPI | 异步 HTTP 服务 |
| AI 编排 | LangGraph | 有状态 DAG 工作流引擎 |
| LLM 调用 | OpenAI SDK | 统一接口，5 家供应商 |
| HTTP 客户端 | httpx | 异步 GitHub API 调用 |
| 数据校验 | Pydantic v2 | 请求/响应模型 |
| 环境管理 | python-dotenv | .env 文件加载 |

### 3.2 应用层 (`app/`)

| 文件 | 职责 |
|------|------|
| `main.py` | FastAPI 实例创建，CORS 中间件配置（允许 localhost:5173），路由注册 |
| `router.py` | `POST /api/analyze-pr` 接口实现：接收请求 → 构建初始状态 → 运行 LangGraph → 映射响应 |
| `schemas.py` | Pydantic 数据模型，与前端 `types.ts` **一一对应** |

### 3.3 工作流层 (`graph/`)

**state.py** — 定义 [`PRAnalysisState`](TypedDict)，工作流中所有节点共享的全局状态：

```
输入字段：pr_url, sandbox_files, template_name
产出字段：repo, pr_title, author, changed_files, summary, risks, suggestions
控制字段：error, current_step
```

**workflow.py** — 构建 DAG 工作流图：

```
parse_pr  →  fetch_diff  →  summarize
                          →  analyze_risks        (fan-out 并发)
                          →  generate_suggestions
```

**nodes.py** — 5 个节点：

| 节点 | 功能 | 关键实现 |
|------|------|----------|
| `parse_pr_node` | 解析 PR URL / 沙盒输入 | 调用 `github.parse_pr_url` |
| `fetch_diff_node` | 获取变更文件内容 | 调用 `github.fetch_pr_files` |
| `summarize_node` | 生成 PR 中文摘要 | 调用 `llm.generate_summary` |
| `analyze_risks_node` | 扫描代码安全风险 | `asyncio.gather` 多文件并发 |
| `generate_suggestions_node` | 生成改进建议 | `asyncio.gather` 多文件并发 |

> ⚠️ **重要规范**：fan-out 并发节点必须只返回自己修改的字段（增量），不能返回完整 state，否则 LangGraph 会报 `INVALID_CONCURRENT_GRAPH_UPDATE`。

**prompts.py** — 3 个 LLM Prompt 模板（`SUMMARY_PROMPT`、`RISK_ANALYSIS_PROMPT`、`SUGGESTION_PROMPT`），使用 `{code_diff}` / `{code_content}` / `{filename}` 占位符。

### 3.4 工具层 (`graph/tool/`)

**github.py** — GitHub API 封装：
- `parse_pr_url(url)` → 正则提取 owner/repo/pr_number
- `fetch_pr_info(owner, repo, pr_number)` → GET /repos/{owner}/{repo}/pulls/{pr_number}
- `fetch_pr_files(owner, repo, pr_number)` → GET pulls/{pr_number}/files + 原始文件内容

**llm.py** — 通用 LLM 调用模块，通过 `LLM_PROVIDER` 环境变量切换供应商：

| 供应商 | 模型示例 | 环境变量 |
|--------|----------|----------|
| DeepSeek (默认) | deepseek-chat | `DEEPSEEK_API_KEY` |
| 通义千问 | qwen-plus | `QWEN_API_KEY` |
| 智谱 GLM | glm-4-flash | `ZHIPU_API_KEY` |
| 月之暗面 Kimi | moonshot-v1-8k | `MOONSHOT_API_KEY` |
| OpenAI | gpt-4o | `OPENAI_API_KEY` |

> 切换供应商只需修改 `.env` 中 `LLM_PROVIDER=xxx`，代码无需任何改动。


```
        │
        ▼
  App.tsx → POST /api/analyze-pr
        │
        ▼
  router.py → 构建 PRAnalysisState
        │
        ▼

    1. parse_pr       → repo, pr_title, author
    2. fetch_diff     → changed_files, files_count
    3. summarize      → summary          ┐
    4. analyze_risks  → risks            ├─ fan-out 并发

        │
        ▼
  App.tsx → 渲染审查报告 UI
```

## 6. 关键设计模式

### 6.1 串行 + 并发混合工作流

前两个节点（parse → fetch）有依赖关系，必须串行执行。后三个 AI 节点（summarize / analyze / suggest）互不依赖，通过 LangGraph fan-out 机制并发执行，大幅缩短总延迟。

### 6.2 asyncio.gather 文件级并发

在 `analyze_risks_node` 和 `generate_suggestions_node` 内部，对多个变更文件使用 `asyncio.gather` 并发调用 LLM，避免逐文件串行等待。

### 6.3 LLM 供应商抽象

通过 `LLM_PROVIDER` 环境变量 + `PROVIDER_CONFIG` 字典实现供应商切换。所有供应商复用 OpenAI SDK（兼容协议），核心接口 `_chat(prompt)` 对外透明。

### 6.4 前后端类型同步

前端 `types.ts` 和后端 `schemas.py` 定义完全一致的数据结构，通过 API JSON 序列化/反序列化保证类型安全。

### 6.5 环境变量安全管理

- `.env` 包含真实密钥，被 `.gitignore` 忽略
- `.env.example` 只含模板占位符，提交到 Git 供协作者参考
- `LLM_PROVIDER` 控制供应商切换，`*_API_KEY` 存储密钥

## 8. 本地开发启动

```bash
# 终端 1：启动后端（需要 conda 环境 pr_assistant）
cd backend
PYTHONUTF8=1 langgraph dev --port 8000

cd frontend
npm run dev
```
